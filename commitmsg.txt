fix: do not create duplicate open follow-up rows
FU-008 and FU-009 were identical rows for the same task -- the same Slack
message got processed twice and _detect_and_log_followup appended blindly.
- followup_update.find_duplicate_open_row(person, task): exact case-insensitive,
  whitespace-stripped match against that person open rows. No fuzzy matching:
  silently merging two genuinely different tasks is worse than one extra row.
- commit_write: check for a duplicate before appending. If one exists, tell the
  user which FU id already tracks it instead of creating a second row.
- closed (done) rows are ignored, so re-committing to a finished task correctly
  opens a new row.
- fix a latent bug: parsed.get("task", "unspecified") returned literal None when
  the LLM sent "task": null, writing None into the table. Now uses `or`.
